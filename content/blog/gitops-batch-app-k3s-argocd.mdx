## 왜 GitOps인가

금융 데이터 스크래핑 배치 시스템(batch-app)은 Docker Compose로 운영하고 있었습니다.
배포는 SSH 접속 → `docker pull` → `restart`의 수동 과정이었고, 어떤 버전이 배포되어 있는지 확인하려면 서버에 직접 접속해야 했습니다.

```
# Before: 수동 배포 흐름
git push → SSH 접속 → docker pull → docker-compose restart → 기도
```

GitOps를 적용하면 **Git이 유일한 진실의 원천**이 됩니다.
코드를 push하면 나머지는 자동으로 일어나고, `git revert` 한 번으로 롤백이 완료됩니다.

```
# After: GitOps 배포 흐름
git push → GitHub Actions → Docker 이미지 빌드 → ArgoCD 자동 감지 → K8s 배포
```

## 아키텍처 설계

### 전체 구조

batch-app의 GitOps 파이프라인은 3개 레이어로 구성했습니다.

| 레이어 | 도구 | 역할 |
|--------|------|------|
| CI | GitHub Actions | 테스트 → 빌드 → Docker 이미지 push |
| CD | ArgoCD | Git 변경 감지 → K8s 자동 동기화 |
| Runtime | k3s (Kubernetes) | 컨테이너 오케스트레이션 |

### 디렉토리 구조

```
batch-app/
├── .github/workflows/ci-cd.yml    # CI/CD 파이프라인
├── k8s/
│   ├── base/                       # 공통 매니페스트
│   │   ├── kustomization.yaml
│   │   ├── deployment.yaml
│   │   ├── service.yaml
│   │   ├── cronjob.yaml
│   │   └── configmap.yaml
│   ├── overlays/
│   │   ├── staging/                # develop 브랜치 → staging 환경
│   │   └── production/             # main 브랜치 → production 환경
│   └── argocd/
│       ├── application.yaml        # production ArgoCD 앱
│       └── application-staging.yaml
├── scripts/k3s-setup.sh            # 원클릭 설치 스크립트
└── Dockerfile                      # 멀티스테이지 빌드
```

## Kustomize: base/overlays 패턴

환경별 설정 차이를 관리하기 위해 Kustomize의 오버레이 패턴을 사용했습니다.

### base — 공통 정의

모든 환경에 공통으로 적용되는 리소스를 정의합니다.

```yaml
# k8s/base/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: batch-app
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: batch-app
          image: ghcr.io/youngmok/batch-app:latest
          ports:
            - containerPort: 9090
          envFrom:
            - configMapRef:
                name: batch-app-config
            - secretRef:
                name: batch-app-secrets
          readinessProbe:
            httpGet:
              path: /actuator/health
              port: 9090
            initialDelaySeconds: 30
          livenessProbe:
            httpGet:
              path: /actuator/health
              port: 9090
            initialDelaySeconds: 60
```

Spring Boot Actuator를 추가하여 K8s health check를 연동했습니다.

### overlays — 환경별 오버라이드

```yaml
# k8s/overlays/staging/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: batch-app-staging
resources:
  - ../../base
namePrefix: staging-
patches:
  - target:
      kind: Deployment
      name: batch-app
    patch: |
      - op: replace
        path: /spec/template/spec/containers/0/resources/limits/memory
        value: "512Mi"
```

staging에서는 메모리를 절반으로 줄이고, production에서는 1Gi를 할당합니다.
같은 base에서 출발하되, 환경별로 필요한 부분만 패치하는 방식입니다.

## GitHub Actions CI/CD 파이프라인

3단계 파이프라인을 구성했습니다.

### 1단계: 테스트

```yaml
test:
  runs-on: ubuntu-latest
  services:
    postgres:
      image: postgres:16-alpine
      env:
        POSTGRES_DB: financial_batch
```

PostgreSQL 서비스 컨테이너를 띄워 실제 DB 연동 테스트까지 수행합니다.
batch-app의 107개 테스트가 모두 통과해야 다음 단계로 진행됩니다.

### 2단계: Docker 이미지 빌드 & Push

```yaml
build-and-push:
  needs: test
  steps:
    - name: Build and push Docker image
      uses: docker/build-push-action@v6
      with:
        push: true
        tags: |
          ghcr.io/youngmok/batch-app:latest
          ghcr.io/youngmok/batch-app:sha-${{ github.sha }}
```

이미지 태그에 `sha-xxxxx`를 사용하여 어떤 커밋이 배포되었는지 추적할 수 있습니다.

### 3단계: 매니페스트 업데이트

```yaml
update-manifest:
  needs: build-and-push
  steps:
    - name: Update image tag in kustomization
      run: |
        cd k8s/overlays/production
        kustomize edit set image \
          ghcr.io/youngmok/batch-app=ghcr.io/youngmok/batch-app:sha-${{ github.sha }}
    - name: Commit and push
      run: |
        git commit -am "chore: update image tag to sha-${{ github.sha }}"
        git push
```

이미지 태그를 Kustomization에 업데이트하고 push하면, ArgoCD가 변경을 감지합니다.

## ArgoCD 자동 동기화

### Application 정의

```yaml
# k8s/argocd/application.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: batch-app
  namespace: argocd
spec:
  source:
    repoURL: https://github.com/youngmok/batch-app.git
    targetRevision: main
    path: k8s/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: batch-app
  syncPolicy:
    automated:
      prune: true      # Git에 없는 리소스 삭제
      selfHeal: true   # 수동 변경 자동 복원
```

핵심은 `selfHeal: true`입니다. 누군가 `kubectl edit`으로 직접 설정을 바꿔도,
ArgoCD가 Git 상태로 **자동 복원**합니다. Git이 곧 운영 환경입니다.

## CronJob으로 배치 스케줄링

기존에는 Spring의 `@Scheduled`로 매일 08시에 배치를 실행했습니다.
K8s 환경에서는 CronJob으로 분리하여 관심사를 분리했습니다.

```yaml
# k8s/base/cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: batch-app-scheduler
spec:
  schedule: "0 8 * * *"
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      backoffLimit: 3
      template:
        spec:
          containers:
            - name: batch-trigger
              image: curlimages/curl:latest
              command:
                - /bin/sh
                - -c
                - |
                  curl -sf -X POST \
                    http://batch-app.batch-app.svc.cluster.local:9090/api/job/run
```

CronJob이 curl로 `/api/job/run`을 호출하는 구조입니다.
`concurrencyPolicy: Forbid`로 이전 Job이 완료되기 전에 새 Job이 시작되지 않도록 보장합니다.

## k3s 원클릭 설치

로컬 환경에서 빠르게 GitOps를 경험할 수 있도록 설치 스크립트를 작성했습니다.

```bash
# scripts/k3s-setup.sh 주요 단계
# 1. k3s 설치 (경량 Kubernetes)
curl -sfL https://get.k3s.io | sh -

# 2. ArgoCD 설치
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/v2.13.3/manifests/install.yaml

# 3. PostgreSQL in-cluster 배포
kubectl apply -f postgresql-deployment.yaml

# 4. ArgoCD Application 등록
kubectl apply -f k8s/argocd/application.yaml
```

스크립트 하나로 k3s → ArgoCD → PostgreSQL → 앱 등록까지 자동 완료됩니다.

## 삽질과 해결

### Kustomize commonLabels deprecated

```
Warning: 'commonLabels' is deprecated. Please use 'labels' instead.
```

최신 Kustomize에서 `commonLabels`가 deprecated되어 `labels` 형식으로 변경했습니다.
`includeSelectors: false`로 설정하면 Deployment의 `matchLabels`와 충돌하지 않습니다.

```yaml
# Before
commonLabels:
  app.kubernetes.io/name: batch-app

# After
labels:
  - pairs:
      app.kubernetes.io/name: batch-app
    includeSelectors: false
```

### Actuator 의존성 누락

K8s의 readiness/liveness probe가 `/actuator/health`를 호출하는데,
Actuator 의존성이 없었습니다. `build.gradle`에 추가 후 107개 테스트가 모두 통과하는 것을 확인했습니다.

```groovy
implementation 'org.springframework.boot:spring-boot-starter-actuator'
```

## 배포 흐름 요약

```
개발자: git push main
         ↓
GitHub Actions:
  ① ./gradlew test (107개 테스트)
  ② docker build → ghcr.io/youngmok/batch-app:sha-xxxxx
  ③ kustomization.yaml newTag 업데이트 → git push
         ↓
ArgoCD (3분 이내 감지):
  ④ Git diff 감지 → kustomize build
  ⑤ kubectl apply → Pod 롤링 업데이트
         ↓
Kubernetes:
  ⑥ 새 Pod → readinessProbe 통과 → 트래픽 전환
  ⑦ CronJob: 매일 08시 배치 실행
         ↓
롤백 필요 시: git revert → 자동 복원
```

## 느낀 점

Docker Compose에서 k3s + ArgoCD로 전환하면서 가장 크게 느낀 것은
**"서버에 접속할 일이 없다"**는 점이었습니다.

Git 리포지토리만 보면 현재 운영 상태를 알 수 있고,
문제가 생기면 `git revert`로 롤백할 수 있습니다.
Self-healing으로 수동 변경도 자동 복원됩니다.

물론 로컬 개발 환경에서는 여전히 Docker Compose가 편리하지만,
운영 환경의 배포 안정성과 추적 가능성은 GitOps가 압도적입니다.

Claude Code와 함께 12개 파일, 566줄의 GitOps 인프라를 한 세션에서 완성했습니다.
AI 없이 했다면 Kustomize 문법, ArgoCD 설정, GitHub Actions 연동 각각에 상당한 시간이 걸렸을 것입니다.
